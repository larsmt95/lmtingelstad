Strava Analytics Pipeline

API-integrasjon, datamodellering og analyse av treningsdata

Problemstilling

Treningsdata fra apper og manuelle logger lagres ofte i separate systemer og i ulike formater. Dette gjør det vanskelig å analysere treningsbelastning, utvikling over tid og prestasjonsmønstre.

Målet med prosjektet var å bygge en automatisert pipeline som:

  • henter aktivitetsdata fra Strava API
  • strukturerer data i en database
  • modellerer datasettet for analyse
  • visualiserer innsikt i Power BI

Executive summary

  • Bygget en automatisert data pipeline (Strava API → R → PostgreSQL → Power BI)
  • Strukturert treningsdata i et stjerneskjema for analyse
  • Utviklet dashboards for analyse av treningsbelastning, prestasjon og treningsmønster

Prosjektet demonstrerer en komplett analysepipeline fra datainnhenting til visualisering.

Data pipeline

Arkitektur for løsningen:

Strava API → Datainnhenting (R) → Databasenivå (PostgreSQL) → Datamodell (Star schema) → Analyse (DAX / Power BI)

Pipeline gjør det mulig å:

  • automatisere datainnhenting
  • lagre historiske treningsdata
  • analysere utvikling over tid

Datarydding i Power Query

Hovedsteg:

  1. Standardisere kolonnenavn og datatyper
  2. Transformere treningsdagbok fra bredt til langt format
  3. Standardisere øktnavn
  4. Fjerne tomme og ugyldige rader
  5. Kombinere årstabeller til én faktatabell
Ryddetrinn (Power Query): Kroppsdel 
Kroppsdel =
  VAR x = UPPER ( 'facts_styrke'[Økt] )
  RETURN
  SWITCH (
    TRUE (),
    CONTAINSSTRING(x, "UB") || CONTAINSSTRING (x, "UPPER") || CONTAINSSTRING (x,          "OVER"),
    "Overkropp",
    CONTAINSSTRING (x, "LB" ) || CONTAINSSTRING (x, "BEIN" ) || CONTAINSSTRING (x,        "LOWER") || CONTAINSSTRING (x, "REHAB"),
    "Underkropp",
    "Ukjent"
)

Resultat etter datarydding

  • én konsistent faktatabell for styrketrening

  • separate dimensjoner for dato og aktivitet

  • datagrunnlag egnet for DAX og tidsserieanalyse

Datainnhenting – Strava API (R)

Strava-data ble hentet via Strava REST API med OAuth-autentisering.

Data hentes i R og lagres i PostgreSQL før analyse i Power BI.

1️⃣ Autentisering (OAuth)

Tilgangstoken hentes via Strava sin OAuth-flyt.
Sensitiv informasjon håndteres via miljøvariabler.

R – Autentisering mot Strava API 
library(httr) 
library(jsonlite)  

res <- POST("https://www.strava.com/oauth/token",   
        body = list(client_id = Sys.getenv("STRAVA_CLIENT_ID"), 
        client_secret = Sys.getenv("STRAVA_CLIENT_SECRET"),
        code = Sys.getenv("STRAVA_AUTH_CODE"),
        grant_type = "authorization_code"),
        encode = "form")
        
tokens <- content(res, "parsed") access_token <- tokens$access_token

2️⃣ Hente aktiviteter

R – Nedlasting av Strava-aktiviteter 
library(dplyr)  

get_activities <- function(page, token) {   
  GET("https://www.strava.com/api/v3/athlete/activities",
  query = list(per_page = 200, page = page), 
  add_headers(Authorization = paste("Bearer", token))
  ) 
  }  
  
all_activities <- list() 
page <- 1  

repeat {
  res  <- get_activities(page, access_token)
  data <- fromJSON(content(res, "text"), flatten = TRUE) 
  
  if (nrow(data) == 0) break  
  
  all_activities[[page]] <- data   
  page <- page + 1 
}  

activities <- bind_rows(all_activities)

3️⃣ Klargjøring av data (API → analyse)

Nested JSON-felter flates ut og reduseres til analyse­relevante kolonner.

activities_sql <- activities %>%   select(

  max_speed,
  average_watts,
  max_watts,
  average_heartrate,
  max_heartrate,
  location_city,
  location_country,
  athlete.id
  )

4️⃣ Lagring i PostgreSQL

Data lagres i en dedikert database for videre bruk i Power BI.

R – Lagring i PostgreSQL 
library(DBI) 
library(RPostgres)

con <- dbConnect(
  Postgres(),
  dbname = "strava",
  host = "xxx",
  port = xxx,
  user = Sys.getenv("PG_USER"),
  password = Sys.getenv("PG_PASSWORD")
  )  
  
dbWriteTable(
  con,
  name = Id(schema = "public", table = "activities"),
  value = activities_sql,
  overwrite = TRUE
)
